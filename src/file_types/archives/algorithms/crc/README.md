# CRC

## 参考

- [维基百科](https://en.wikipedia.org/wiki/Cyclic_redundancy_check)

## 介绍

CRC（Cyclic Redundancy Check）是一种常用的错误检测码，用于检测一段可以使用数字形式表示的数据的意外变化。该码也是一种 Hash，因此 CRC 计算过程是一种 Hash 函数。CRC 通常会被附在需要校验的数据后面，接收方通常需要用相同算法进行一次计算，通过判断余数是否为 0 来确认原始数据是否存在意外变化。

CRC 之所以叫这个名字，因为 CRC 本身是一种冗余（额外附加在原始数据之外的校验数据，不影响原始数据的内容），同时计算过程使用了循环码（Cyclic Code，多轮计算，每一轮的结果进行位移后会作为下一轮的输入）。

CRC 根据附加的长度 n 可以称为 CRC-n。比较常用的包括 CRC-8，CRC-16，CRC-32，CRC-64 等。通常而言，n 位 CRC 可以发现原始数据中最大 n 位的意外改变。

### 优势

1. 简单，效率可接受，有很多种优化方式，甚至有一些硬件手段支持
2. 流式友好，可以在处理流式数据时线性计算
3. 比起 checksum 而言更容易发现数据问题（没有 checksum 多个错误可能互相抵消等问题）

### 劣势

1. 在极端场景可能有性能问题（不多见）
2. 只负责发现“存在错误”，但是无法发现“哪里存在错误”，也无法修复错误
3. 不抗篡改：CRC 不是一种签名，如果想篡改，随时可以修改原始数据后重新计算 CRC
4. 不抗伪造：可以通过系统方式对原始数据进行篡改并且使得最终 CRC 计算值保持不变；所以通常仅用于对抗随机错误，如网络传输等

## 算法思路

### 理论背景

由于本人数学不是很好，所以只会用最简单的语言阐述一些保证是正确的内容，更复杂的部分可以自行查阅相关资料。

CRC 计算的核心思想可表述如下：将元数据转换成二进制表示，并将其视为 GF(2) 上的多项式系数，作为被除数；同时给定一个系数定义在 GF(2) 上的多项式，作为除数，进行多项式除法。该除法最终计算会得到一个商和一个余数，商被抛弃，而该余数也会是一个多项式；该余数提取系数便为 CRC 计算的结果。

下面一项一项来解释。

#### GF(2)

GF(2) 是具有两个元素的有限域。GF 就是有限域，其字母为伽罗华域（Galois Field）的缩写，该名字来源于对法国数学家 Evariste Galois 的纪念。

在 GF(2) 中，这两个元素通常被表计为 0 与 1。在该域上进行的计算与我们熟知的布尔代数基本一致，我们需要关注的是以下两个算式：

```
1 + 1 = 0
0 - 1 = 1
```

因此在该域上的减法与加法完全相同，同时又都可以规约成异或运算。

#### 多项式除法

大多数人应该在高中阶段就系统性的学习过，这里就不过多赘述，简单讲一下概念。
多项式除法就是将一个多项式作为被除数，另一个多项式作为除数，最终得出一个商式与一个余数式的过程。该过程通常通过列竖式进行计算，与一般的除法竖式相似，该过程会将除法转化为乘法与减法计算。

以下为一个例子（来自<https://zh.wikipedia.org/wiki/%E5%A4%9A%E9%A1%B9%E5%BC%8F%E9%99%A4%E6%B3%95>）：

$$
\begin{matrix}
\ x^2 - 9x \quad - 27\\
x-3\overline{) x^3 - 12x^2 + 0x - 42}\\
\underline{\;\;x^3 - \;\;3x^2\qquad \;}\\
\quad \;\quad-\;\; 9x^{2}+0x \;\\
\quad \;\;\quad\underline{-\;\; 9x^2 + 27x}\;\\
\qquad\qquad\qquad\qquad-27x - 42 \\
\qquad\qquad\qquad\qquad\underline{-27x + 81}\\
\qquad\qquad\qquad\qquad\qquad\;\; -123
\end{matrix}
$$

上式中，最上方横线之上的便为商，剩下的（-123）即为余数。

$$
\frac{x^3 - 12x^2 - 42}{x-3} = \underbrace{x^2 - 9x - 27}_{q(x)}  \underbrace{-\frac{123}{x-3}}_{r(x)/g(x)}
$$

在不会混淆的情况下，有时可以省略到式子中所有的 $x$ 与上标，只保留系数。特别的，在 GF(2) 下运算时，因为既不会出现大于 1 的数也不会出现负数，因此将所有的系数全部拼在一起写也不会产生歧义。

### 算法流程

上面有提到，计算 CRC 的过程就是在做多项式除法。我们需要进行计算的原串转换为二进制编码，每一位对应一个多项式系数，就可以得到被除数。而除数怎么取呢？在不同的 CRC 规范中除数的取法各有千秋，这个后面再说。

首先，CRC-n 需要 n+1 位作为除数。以 <https://en.wikipedia.org/wiki/Cyclic_redundancy_check> 中的一个例子来举例，我们来计算一下 CRC-3。CRC-3 中除数的最高位为 $x^3$。我们先约定一个除数，假设为 $x^3 + x + 1$。然后我们将其转化为系数表示，为 1011（$1x^3 + 0x^2 + 1x + 1$）。我们取需要编码的串，转化为二进制表示，假设为如下内容

```
11010011101100
```

这个数我们也要理解里面每一个 1 和 0 其实都是一个多项式系数。为了让 CRC 便于校验，通常的做法是在计算时在被除数后面补 n 个 0，这样可以保证计算出来的余数部分不会包含被除数，同时可以直接将余数拼接在原被除数后进行发送。这样的好处是，在校验时可以像计算时一样顺序执行一遍多项式除法，如果余数为 0 则可确认校验通过。补完零后，被除数会变成这样（数据中的空格只是为了示意哪部分是原始数据）：

```
11010011101100 000
```

然后我们执行完整的计算过程：

```
11010011101100 000 <--- input padded by 3 bits from the right
1011               <--- divisor
01100011101100 000 <--- result (the first four bits are the XOR with the divisor beneath, the rest of the bits are unchanged)
 1011              <--- divisor ...
00111011101100 000
  1011
00010111101100 000
   1011
00000001101100 000 <--- the divisor moves over to align with the next 1 in the dividend (since quotient for that step was zero)
       1011             (in other words, it doesn't necessarily move one bit per iteration)
00000000110100 000
        1011
00000000011000 000
         1011
00000000001110 000
          1011
00000000000101 000
           101 1
-----------------
00000000000000 100 <--- remainder (3 bits).  Division algorithm stops here as dividend is equal to zero.
```

整个计算过程与正常的多项式除法无异，但是由于 GF(2) 的特殊性，可以看到减法相当于在做按位异或。

根据上面的计算结果，我们将商抛弃（也没画在上面的图上，根本不用不着），留下余数 100，它就是这个 CRC-3 的计算结果。发送时，我们将该结果附在原始数据之后，就像这样：

```
11010011101100 100
```

校验方则使用相同的除数进行除法动作：

```
11010011101100 100 <--- input with check value
1011               <--- divisor
01100011101100 100 <--- result
 1011              <--- divisor ...
00111011101100 100

......

00000000001110 100
          1011
00000000000101 100
           101 1
------------------
00000000000000 000 <--- remainder
```

余数为 0 则校验通过。

## 规范

### 多项式的表示方式

CRC 只是一种算法或思想，在具体实现时则产生了多种不同的规范。其中最麻烦的便是多项式的表示方式。比方说一个多项式 $x^4 + x + 1$，虽然我们在直觉上认为它应该被表示为 0b10011，但是在实践时时出现了以下几种不同的表示方式，并且每种表示方式都有意的忽略了其中一项，导致编码时只会存在 4 个二进制位：

- 0x3 = 0b0011，表示 $1x^4 + (0x^3 + 0x^2 + 1x^1 + 1x^0)$，忽略了最高位，称为正常（Normal）表示，在最高有效位（MSB(Most-Significant Byte)-first code）模型中使用
- 0xC = 0b1100，表示 $(1x^0 + 1x^1 + 0x^2 + 0x^3) + 1x^4$，依然忽略了最高位，但是与正常表示相反，称为反转（Reversed）表示，在最低有效位（LSB-first code）模型中使用
- 0x9 = 0b1001，表示 $(1x^4 + 0x^3 + 0x^2 + 1x^1) + 1x^0$，忽略了最低位（常数位），称为 Reversed reciprocal 表示，在 Koopman notation 模型中使用，较少见

### 多项式的取值

不同的位数、不同的规范下多项式取值都不相同，其中 CRC-32/IEEE 使用的多项式如下：

$$
x^{32}+x^{26}+x^{23}+x^{22}+x^{16}+x^{12}+x^{11}+x^{10}+x^{8}+x^{7}+x^{5}+x^{4}+x^{2}+x+1
$$

### 其他

不同的规范还定义了一些奇怪的东西，比如说有的规范会对输入内容或者 CRC 值进行位翻转，又或者对结果与一个常量进行异或等。所以在实践时必须严格遵守所使用的 CRC 规范，否则计算结果就会出错。
