## 介绍

LZ77 是 Abraham Lempel 与 Jacob Ziv 在 1977 年提出的一种无损压缩算法。在 1978 年他们改进了该算法并提出了 LZ78。LZ77 与 LZ78 是许多现代压缩算法的基础，并且也产生了很多变体，如 LZW、LZSS、LZMA 等。同时这两种算法也被作为很多现代压缩方案的算法基础，如基于 LZ77 与哈夫曼算法广泛存在与 PNG、GZIP 等格式中。

### 优势

1. 无损压缩，压了可以还原
2. 通用压缩方式，对源文件没有要求
3. 编码速度快，复杂度低

### 劣势

1. 强依赖对文件重复模式的发现，如果文件重复模式少或者窗口大小不合理，会极大影响压缩率
2. 由于对于源文件没有感知，对于有特定编码方式但是在广义二进制上看起来像是随机数据的文件压缩率低（如 mp3 等）

## 算法思路

### 编码

一言以蔽之，就是使用一个滑动窗口划过需要加密的串，每滑动一次就将原串的一个子串编码为一个固定长度的串。在编码时，通过前面已知的串找到与需要编码的串的最长公共子串，在编码中记录公共子串在已知串中的起点，这样在解码时就可以通过已经解码的串将待解码的串还原回来。

滑动窗口被分割为两个部分，分别为前面的**搜索区（search buffer）**，以及后面的**前向缓冲区（look-ahead buffer）**，或称之为等待编码区。编码动作可以用下面的过程描述：

0. 将串的每一位作为 α 进制的数字来看待。比如串是由二进制构成，则可以看作每一位为 256 进制的数字。滑动窗口的长度设为 `n`，其中前向缓冲区的长度记为 `Ls`。
1. 起始时，搜索区被置为全 0，前向缓冲区被置为原串 `S` 的头部。滑动窗口中的串记为 `B`。
2. 记前向缓冲区的起始点为 `j`。在搜索区中查找一个索引 `p`，使得滑动窗口中的子串 `B[p, p + l]` 与 `B[j, j + l]` 相同，且 `l` 达到最大（最长公共子串），且 `B[j, j + l]` 不覆盖滑动窗口的最后一个字符（保留字符）。
3. 对子串 `[j, j + l + 1]` 进行编码。编码规则如下：
   3.1. 取 `C1 = p`，转化为 α 为底的数字，再转为与原串相同的一位表示，位数补齐到 $\lceil\log_{|A|}(n - L_s)\rceil$ 位，前补 0
   3.2. 取 `C2 = l`，转化为 α 为底的数字，再转为与原串相同的一位表示，位数补齐到 $\lceil\log_{|A|}(L_s)\rceil$ 位，前补 0
   3.3. 取 `C3 = s = B[j + l + 1]`
   3.4. 取 `C=C1C2C3`，将三个数字拼合在一起，最终获得一个长为 $\lceil \log_{|A|}(n - L_s) \rceil + \lceil \log_{|A|} L_s \rceil + 1$ 的串
4. 将滑动窗口向右移动 `l + 1` 位，重复 2 ～ 4 的动作，直到前向缓冲区不满。
5. 将每一次生成的 `C` 拼在一起，作为编码串；最终遗留在前向缓冲区中的串记为剩余串 `R`，结束编码。

### 解码

解码只需要简单将编码的动作反过来就行了。此处编码顺序与解码顺序相同，都是从左向右（有点难说明白，可以自行感受一下）。

1. 起始时，搜索区全置为 0，前向缓冲区在解码时用不着，方便起见也可以全置为 0。
2. 从编码串中取出一个 `C`，解析出`C1`、`C2`、`C3`，进一步解析出 `p`、`l`、`s`。
3. 根据 `p` 与 `l`，可以在前向缓冲区中根据搜索区的内容写出长为 `l` 的串。
4. 将 `s` 拼接到串最后，则此时前向缓冲区存在长为 `l + 1` 的串。
5. 向右滑动窗口，直至前向缓冲区为空。重复 2 ～ 5 的动作，直至编码串全部消耗完。
6. 将过程中拼接出来的串取出，在后面拼接上剩余串 `R`，剔除步骤 1 插入的的 `0`，则得到还原后的原串。
